# ADR 0008: Language Frontends (Reverse Transpilation)

- **Status:** Accepted
- **Date:** 2026-07-10
- **Deciders:** UEAS Architectural Review
- **Supersedes:** None
- **Superseded By:** None

## Context

UEAS transpiles algorithms from its Canonical JSON AST out to target languages (Python, Java, Rust) via the `backends/` domain. However, a significant barrier to adoption is the requirement for users to rewrite existing, working algorithms into UEAS pseudocode to access the standard's complexity profiling and formal verification features.

## Decision

We will implement "Reverse Transpilation" by introducing a new bounded context: `frontends/`.
This domain will house language-specific extractors (e.g., `frontends/python/`) that parse source code from target languages and map it *inward* to the UEAS Canonical JSON AST. 

This establishes UEAS as a **Universal Algorithm Extractor**. Once an algorithm is converted into the Canonical AST, it can be seamlessly passed to the `kernel/` for abstract interpretation or to `backends/lean.rs` for formal verification, effectively bringing mathematical rigor to existing codebases.

## Consequences

**Positive:**
- **Massive Adoption Catalyst:** Users can verify existing Python/C++ code without learning UEAS pseudocode syntax.
- **Architectural Purity:** The `kernel/` and `backends/` are completely untouched. They only consume the Canonical AST and do not care if it was generated by `UEAS.g4` or `frontends/python`.
- **Auto-Complexity:** The frontend can perform static analysis to deduce polynomial complexity bounds if the user hasn't provided a strict contract.

**Negative:**
- **Axiom Enforcement Friction:** Target languages allow system I/O (e.g., `print()`, file access, network calls) and memory aliasing (e.g., raw pointers in C++). The frontends MUST perform aggressive semantic analysis to reject any code that violates UEAS Axiom 1 (Zero System I/O) and the Single-Ownership memory model. This makes the frontends significantly more complex than the backends.

## Alternatives Considered

1. **Silent I/O Stripping:** We considered allowing the frontend to quietly drop `print()` statements or I/O calls to extract just the algorithmic logic. **Rejected.** This violates the Semantic Equivalence Guarantee. If an algorithm depends on the side-effects of an I/O call, stripping it changes the behavior. The frontend must **Hard Reject** any source code that violates UEAS axioms. Users must isolate their algorithmic logic from their I/O logic before extracting it.
